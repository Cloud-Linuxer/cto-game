# AWS 스타트업 타이쿤 - 게임 시스템 설계서

---

## 1. 게임 개요

턴 기반 경영 시뮬레이션. 플레이어는 스타트업 CTO로서 비즈니스 결정과 AWS 인프라 설계를 동시에 수행하며, EC2 하나에서 시작해 IPO를 목표로 성장합니다.

| 항목 | 내용 |
|------|------|
| 장르 | 턴 기반 전략 시뮬레이션 |
| 턴 수 | 22~30턴 (난이도별) |
| 선택지 | 253개 (턴당 5~7개) |
| 승리 조건 | 4가지 경로 |
| 패배 조건 | 5가지 |

---

## 2. 난이도 시스템

| 장애 임계값 (신뢰도) | 5% | 10% | 15% |
| 점수 배율 | 0.6x | 1.0x | 1.5x |

## 4. 턴 진행 흐름

```
턴 시작
  │
  ├─ 1) 턴 시작 회복 처리
  │     ├─ 신뢰도 < 30% → 자연 회복 (+1~2/턴)
  │     ├─ 자금 < 0 → 부채 이자 5% 부과
  │     └─ 연속 적자 턴 카운터 갱신
  │
  ├─ 2) 이벤트 텍스트 표시
  │
  ├─ 3) 선택지 제시 (5~7개)
  │     └─ 멀티초이스 가능 시 2개 선택
  │
  ├─ 4) 선택 실행
  │     ├─ 투자 조건 확인 (스케일링)
  │     ├─ 인프라 추가 → 수용량 재계산
  │     ├─ 효과 적용 (난이도 배율 × 컴백 배율)
  │     ├─ 수용량 초과 확인 → 단계별 페널티
  │     └─ 직원 채용 효과 처리
  │
  ├─ 5) 상태 확인
  │     ├─ 패배 조건 체크 (파산/장애/지분)
  │     ├─ IPO 조건 충족 → 특수 턴 이동
  │     └─ 긴급 이벤트 트리거 확인
  │
  └─ 6) 다음 턴 이동
        └─ 최종 턴 도달 시 → 승리 경로 판정
```

## 5. 선택지 효과 시스템

### 5.1 기본 효과

각 선택지는 다음 효과를 가짐:

```
effects: {
  users: number    // 유저 수 증감
  cash: number     // 자금 증감
  trust: number    // 신뢰도 증감
  infra: string[]  // 추가될 AWS 서비스
}
```

### 5.2 효과 배율 적용 순서

```
최종 효과 = 기본 효과
           × 난이도 배율 (긍정/부정 별도)
           × 직원 배율 (디자이너→유저, 기획자→신뢰도)
           × 컴백 배율 (위험 시 1.25x, 긍정 효과만)
```

### 5.3 특수 선택지

| 선택지 | 턴 | 효과 |
|--------|-----|------|
| 초기 피칭 (ID 8) | 2 | 신뢰도 미달 시 현재 신뢰도 50% 감소 |
| Series A | 12 | 자금 효과 ≥ 1억 → 신뢰도 비례 스케일링 |
| Series B | 18 | 자금 효과 ≥ 10억 → 신뢰도 비례 스케일링 |
| Series C | 23 | 자금 효과 ≥ 30억 → 신뢰도 비례 스케일링 |
| 컨설팅 (ID 68) | - | 수용량 3배 영구 증가 (1회성) |

### 5.4 투자 스케일링 (차단이 아닌 비례 조정)

```
스케일 = clamp(현재 신뢰도 / 목표 신뢰도, 0.3, 1.5)
실제 투자금 = floor(투자금 × 스케일)
```

- 신뢰도가 부족해도 최소 30%는 투자 유치 가능
- 신뢰도 초과 시 최대 150%까지 보너스

## 6. 인프라 수용량 시스템

### 6.1 가산 방식 (Additive)

```
총 수용량 = 기본 수용량 (5,000)
          + Σ(보유 인프라별 수용량)
          × 컨설팅 배율 (3x, 적용 시)
          × (1 + 복원력 스택 × 5%)
```

### 6.2 인프라별 수용량

| 서비스 | 수용량 | 서비스 | 수용량 |
|--------|--------|--------|--------|
| EC2 | 10,000 | Auto Scaling | 40,000 |
| Route53 | 5,000 | ECS | 30,000 |
| CloudWatch | 5,000 | EKS | 60,000 |
| RDS | 15,000 | Karpenter | 40,000 |
| S3 | 15,000 | Lambda | 40,000 |
| Aurora | 50,000 | Bedrock | 30,000 |
| Redis | 30,000 | Aurora Global DB | 80,000 |
| CloudFront | 50,000 | multi-region | 100,000 |

### 6.3 단계별 초과 페널티

| 초과 비율 | 신뢰도 감소 |
|-----------|------------|
| 10% 이상 | -2 |
| 30% 이상 | -4 |
| 50% 이상 | -6 |
| 100% 이상 | -8 |

수용량 경고: 70% → YELLOW, 90% → RED

## 7. 회복 & 복원력 시스템

### 7.1 자연 신뢰도 회복

| 조건 | 회복량 | 상한 |
|------|--------|------|
| 신뢰도 < 30% | +1/턴 | 30%까지 |
| 신뢰도 < 15% | +2/턴 | 30%까지 |
| 복원력 스택 보유 | +1/스택/턴 추가 | - |

### 7.2 파산 유예 (Grace Period)

- 자금 < 0 이어도 즉시 파산하지 않음
- **3턴 유예** 후 파산 처리
- 유예 중 매턴 부채의 **5% 이자** 부과
- 하드 임계값(NORMAL: -3천만) 초과 시 즉시 파산

### 7.3 복원력 스택

- 인프라 수용량 초과를 생존하면 **+1 스택** (최대 3)
- 스택당 효과: 수용량 +5%, 신뢰도 회복 +1/턴
- 최대 3스택 = 수용량 +15%, 회복 +3/턴

### 7.4 컴백 배율

- 주요 지표가 IPO 목표의 **30% 미만**일 때 활성화
- 긍정적 효과에 **1.25배** 배율 적용
- 유저/자금/신뢰도 중 하나라도 해당하면 활성

## 8. 승리 조건 (4가지 경로)

### 8.1 승리 경로별 조건 (NORMAL 기준)

| 경로 | 유저 | 자금 | 신뢰도 | 인프라 | 점수 배율 |
|------|------|------|--------|--------|----------|
| **IPO 상장** | 80,000+ | ₩200M+ | 65%+ | RDS, EKS 필수 | 1.0x |
| **기술 선도** | 40,000+ | ₩80M+ | 85%+ | 10개+ | 0.9x |
| **인수합병** | 60,000+ | ₩80M+ | 70%+ | 8개+ | 0.85x |
| **흑자 전환** | 30,000+ | ₩500M+ | 50%+ | 제한 없음 | 0.75x |

### 8.2 판정 우선순위

최종 턴 도달 시 순서대로 확인:
**IPO → 기술 선도 → 인수합병 → 흑자 전환**

어떤 경로도 충족하지 못하면 → **LOST_FIRED_CTO** (해고)

### 8.3 IPO 특수 이벤트

- IPO 조건 충족 시 **턴 950**(IPO 선택 화면)으로 이동
- 즉시 IPO 수락 또는 계속 플레이 선택 가능
- 계속 플레이 선택 시 원래 턴으로 복귀

## 12. 점수 계산 & 리더보드

### 12.1 점수 산출

```
유저 점수 = 유저 수 (1명 = 1점)
자금 점수 = floor(자금 / 10,000)
신뢰 점수 = 신뢰도 × 1,000

기본 점수 = 유저 점수 + 자금 점수 + 신뢰 점수
최종 점수 = floor(기본 점수 × 난이도 배율 × 승리 경로 배율)
```

### 12.2 배율 조합 예시

| 시나리오 | 계산 |
|----------|------|
| EASY + 흑자 전환 | 기본 × 0.6 × 0.75 = **0.45x** |
| NORMAL + IPO | 기본 × 1.0 × 1.0 = **1.0x** |
| HARD + IPO | 기본 × 1.5 × 1.0 = **1.5x** |
| HARD + 기술 선도 | 기본 × 1.5 × 0.9 = **1.35x** |

## 15. 게임 밸런스 설계 원칙

1. **차단 대신 스케일링**: 조건 미달 시 차단하지 않고 비례 축소
2. **가산 수용량**: 인프라를 쌓을수록 선형 증가
3. **단계별 페널티**: 이진(ON/OFF)이 아닌 비율 기반 징벌
4. **회복 메커니즘**: 위기 상황에서 자연 회복 경로 제공
5. **복수 승리 경로**: 하나의 목표에 실패해도 대안 존재
6. **난이도별 차등**: 같은 구조에서 수치만 조정하여 난이도 구현
7. **컴백 보너스**: 뒤처진 플레이어에게 추가 기회 부여

#### 턴 2 — 📍 MVP 구축 단계

**상황**: 유저 0~100명, EC2 1대. 첫 투자 기회 등장

| ID | 선택지 | 카테고리 | 유저 | 자금 | 신뢰도 | 인프라 |
|----|--------|----------|------|------|--------|--------|

| 9 | CloudWatch 모니터링 구축 (5만 원) | 인프라 | 0 | -40,000 | +3 | **CloudWatch** |

**⭐ 특수 메커니즘: ID 8 (초기 피칭)**

```
투자 스케일링 적용:
  스케일 = clamp(현재 신뢰도 / 목표 신뢰도, 0.3, 1.5)
  목표 신뢰도: EASY=3, NORMAL=5, HARD=8

  신뢰도 미달 시 → 투자금 비례 감소 + 현재 신뢰도 50% 차감
```

**투자 스케일링 공식**:
```
스케일 = clamp(현재 신뢰도 / 목표 신뢰도, 0.3, 1.5)
실제 투자금 = floor(기본 투자금 × 스케일)

예시: Series A 적정(10억), 신뢰도 50%, 목표 25
  → 스케일 = clamp(50/25, 0.3, 1.5) = 1.5
  → 실제 = floor(10억 × 1.5) = 15억
```

### 17.6 IPO 조건 체크 상세

매 턴 선택 실행 후 자동 체크:

```
조건 1: 유저 ≥ ipoMinUsers     (EASY: 70,000 / NORMAL: 80,000 / HARD: 120,000)
조건 2: 자금 ≥ ipoMinCash      (EASY/NORMAL: ₩200,000,000 / HARD: ₩400,000,000)
조건 3: 신뢰도 ≥ ipoMinTrust   (EASY: 60 / NORMAL: 65 / HARD: 85)
조건 4: 인프라에 RDS 또는 Aurora 포함 AND EKS 포함

4개 모두 충족 → ipoConditionMet = true → 턴 950으로 이동
```

### 17.7 최종 턴 승리 경로 판정

최종 턴(EASY:30, NORMAL:25, HARD:22) 도달 시 순서대로 확인:

```
1순위: IPO 상장    → 유저 80k+ / 자금 ₩200M+ / 신뢰 65%+ / RDS+EKS  → 점수 ×1.0
2순위: 기술 선도   → 유저 40k+ / 자금 ₩80M+  / 신뢰 85%+ / 인프라 10개+ → 점수 ×0.9
3순위: 인수합병    → 유저 60k+ / 자금 ₩80M+  / 신뢰 70%+ / 인프라 8개+  → 점수 ×0.85
4순위: 흑자 전환   → 유저 30k+ / 자금 ₩500M+ / 신뢰 50%+              → 점수 ×0.75
미달:  CTO 해고   → 어떤 경로도 충족 못함                              → LOST_FIRED_CTO
```
